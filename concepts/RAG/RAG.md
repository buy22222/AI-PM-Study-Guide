# RAG架构（检索增强生成）

RAG 全称检索增强生成（Retrieval Augmented Generation），核心含义是通过自有垂域数据库检索相关信息，合并入提示模板供大模型生成回答。

痛点：
通用大模型无法处理实时性的、非公开的或私域的数据

解决问题：
知识局限、幻觉、数据安全

两个阶段：
数据准备、应用阶段

## 数据准备
### 数据提取
1. 数据加载 (Data Loading)：源数据
2. 数据处理 (Data Processing)：数据提取过程包括对数据进行过滤、压缩和格式化
3. 元数据获取 (Metadata Acquisition)：提取关键信息，例如文件名、标题、时间戳等

### 数据准备
提取后的数据会按照以下路径流动：
1. 数据提取： 完成格式统一与清洗。
2. 文本分割 (Chunking)： 将提取的长文本拆分为语义完整的块，以适配 Embedding 模型的 Token 限制
3. 向量化 (Embedding)： 将清洗后的文本块转化为数值矩阵
4. 数据入库： 将向量和元数据一起存入 FAISS、Milvus 等向量数据库中

## 应用
### 查询向量化 (Query Vectorization)
当用户提交提问（Question）时，系统首先需要将自然语言转化为计算机可理解的数学表达。
• 模型一致性： 必须使用与数据准备阶段完全相同的 Transformer Encoder 编码器模型对查询进行向量化。
• 语义映射： 这一步将用户的意图映射到与知识库相同的向量空间中，为后续的语义比对奠定基础。
### 数据检索与召回 (Data Retrieval/Recall)
这是应用阶段的“搜索”环节，旨在从海量索引中找出最相关的知识片段。
• 检索方法： 原始 RAG 主要依赖相似性检索（如计算查询向量与存储向量之间的余弦相似性或欧氏距离），返回得分最高的 Top-K 个结果。
• 多维检索： 除了向量相似性，也可以结合传统的全文检索（基于关键词的倒排索引）来提升召回的覆盖率。
### 注入 Prompt (Prompt Injection)
这是将检索到的“外部知识”与大模型结合的关键一步，体现了 “RAG = 检索技术 + LLM 提示” 的核心公式。
• Prompt 的构成： 系统会将检索到的文本块作为背景知识（Context），连同任务描述和用户提问一起填入提示词模板中。
• 开卷考试模式： 这种设计本质上是给大模型提供了一份“参考资料”，要求它基于这些事实进行回答，从而有效缓解模型的幻觉问题并突破其知识局限性。
### LLM 生成答案 (LLM Generation)
最后，大模型发挥其强大的推理和语言生成能力，输出最终答案。
• 角色定位： 在此阶段，大模型不再是单纯的“知识库”，而是一个**“推理引擎”**。它负责润色、总结并确保回答符合任务指令（如简洁性要求或特定的格式）。
• 生产挑战： LLM 调用通常是整个流程中耗时最长的操作。为了优化用户体验，通常会采用流式输出来缩短感知的首字响应时间。

## RAG优化
### 检索前的增强：查询转换与路由 (Query Transformation & Routing)
为了解决用户提问表述不清或过于复杂的问题，高级 RAG 引入了由 LLM 驱动的预处理机制：
• 查询转换Query Transformation： 利用 LLM 作为推理引擎对输入进行修改。例如，**多查询检索（Multi-Query）**能将一个复杂问题拆解为多个子查询，以覆盖问题的各个方面；**HyDE（假设性响应）**则先由 LLM 生成一个假设答案，再利用该答案的向量进行检索，以提高语义相似性匹配的质量。
• 查询路由Routing： 这是一种 LLM 驱动的决策步骤。路由算法会根据查询内容决定下一步动作，例如是执行摘要、搜索特定索引，还是尝试多个不同路由并综合输出。

### 检索阶段的精细化：混合搜索与索引优化
高级 RAG 不再局限于单一的向量检索，而是通过多种技术的融合提升召回率：
• 融合检索（混合搜索）： 结合了传统的关键词搜索（如 BM25）语义向量搜索。利用 RRF（Reciprocal Rank Fusion）算法对不同来源的搜索结果进行重新排序，这种方式互补了语义匹配和精确关键词匹配的不足。
• 层级与分层索引： 在大型数据库中，可以创建摘要索引和文档块索引。首先通过摘要过滤文档，再在相关组内搜索细节，从而大幅提升海量数据下的检索效率。
• 元数据过滤： 存储向量时附带文件名、时间、Title 等元数据，允许系统按日期或来源等条件进行精准过滤

### 检索后的优化：内容增强与重排 (Context Enrichment & Reranking)
召回后的数据往往需要进一步处理，以适配 LLM 的推理需求：
• 内容增强技术Context Enrichment： 包括语句窗口检索（检索单个句子后扩展其前后的上下文）和自动合并检索（当多个子块被检索到时，自动将其合并为更大的父块），这能平衡检索的精准度与推理所需的语境完整性。
• 重排Reranking： 这是将检索到的上下文提供给 LLM 之前的关键一步。通过使用**交叉编码器（Cross-Encoder）**等模型，根据相似性分数或元数据对初步召回的结果进行二次细化，剔除无关信息

### 逻辑深度：聊天引擎与智能体 (Chat Engine & Agents)
高级 RAG 将 RAG 流程与对话逻辑及智能体架构深度融合：
• 聊天引擎Chat Engine： 引入查询压缩技术来考虑对话上下文。例如 CondensePlusContextMode 会将聊天记录与最后一条消息压缩成新查询后再进入索引，从而支持代词指代和后续追问。
• 多文档智能体（Agent）： 这是 RAG 架构的高级形态。每个文档可以配置一个具备摘要和问答能力的智能体，并由一个顶层智能体负责任务分配与答案综合。这种架构虽然运行速度较慢，但能处理极其复杂的文档集合比对任务。

### 性能提升的底座：微调与 RAG 融合 (Fine-tuning & RAG Fusion)
• 微调（Fine-tuning）： 可以对 Transformer 编码器进行微调以提升嵌入质量，或对 LLM 进行微调以使其更好地利用检索到的上下文。例如 RA-DIT 技术提出了一种同时调整 LLM 和检索器的双重指令微调方法。
• RAG 融合（RAG Fusion）： 通过多查询生成和结果重排，提供多样化的上下文并实现自动校正（如纠正拼写错误），但也面临延迟增加和成本上升的权衡。

### 评估体系：RAG 三元组
为了量化这些高级技术的有效性，行业提出了 RAG 三元组（RAG Triad） 评估模式，分别对以下三个维度进行衡量：
• 检索内容相关性 (Retrieval Context Relevance)： 衡量检索到的背景信息是否真的能回答用户的问题。这是 RAG 系统最关键且最可控的指标。
• 答案基于性/忠实度 (Groundedness/Faithfulness)： 衡量大模型生成的回答在多大程度上是由提供的上下文支持的，用于监测模型是否产生了“幻觉”。
• 答案相关性 (Answer Relevance)： 衡量生成的答案是否直接、准确地回应了用户的初始提问。

#### 具体评估指标分类
• 检索性能指标：
    ◦ 命中率 (Hit Rate)： 衡量相关文档是否出现在返回结果中。
    ◦ 平均倒数排名 (MRR)： 衡量相关结果在返回列表中的排名靠前程度。
    ◦ 精准度 (Precision) 与 召回率 (Recall)： Ragas 等框架使用的经典检索评估指标。
• 生成质量指标：
    ◦ 真实性 (Truthfulness)： 评价生成答案的客观正确性。
    ◦ 忠实度指标（Faithfulness）： 专门用于评估模型利用上下文的能力。

####  评估工具与框架
为了实现上述指标的自动化测量，来源提到了几个主流框架：
• Ragas： 专注于使用真实性和相关性来评价生成质量，并评估检索性能。
• Truelens： 提出了上述的 RAG 三元组评估模式。
• LangSmith (LangChain)： 提供先进的监控和自定义评估器，增强了 RAG 管道的透明度。
• rag_evaluator (LlamaIndex)： 针对 LlamaIndex 用户的专用评估工具包。

## 相关开源生态
• Dify： 被定位为可以超快速构建 AI Agent 的开源平台。
• LlamaIndex 与 LangChain： 两者均提供了对智能体架构（如 ReAct 智能体）的深度支持，让开发者能将 RAG 流程封装进智能体中。
